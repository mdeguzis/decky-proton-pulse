# i18n System Design

## Overview

A lightweight internationalization system for the Decky Proton Pulse plugin supporting 10 languages with automatic detection, runtime key lookup, and compile-time safety. No external dependencies.

## Languages

| Code   | Name        | Native        |
|--------|-------------|---------------|
| en     | English     | English       |
| zh-CN  | Chinese (S) | 简体中文      |
| ru     | Russian     | Русский       |
| pt-BR  | Portuguese  | Português     |
| de     | German      | Deutsch       |
| es     | Spanish     | Español       |
| fr     | French      | Français      |
| ja     | Japanese    | 日本語        |
| ko     | Korean      | 한국어        |
| tr     | Turkish     | Türkçe        |

## Architecture

A single TypeScript module at `src/lib/i18n.ts` exports:

1. **`Language` type** — union of the 10 locale codes (`'en' | 'zh-CN' | 'ru' | ...`)
2. **`TranslationTree` interface** — nested object type with string leaves and function leaves for dynamic keys
3. **Translations record** — `Record<Language, TranslationTree>` containing all translations inline
4. **`detectLanguage(): Language`** — detection chain (see below)
5. **`t()` accessor** — returns the resolved `TranslationTree` for the active language with English fallbacks
6. **`setLanguage(lang: 'auto' | Language)`** — updates preference and triggers re-render
7. **`useLanguage()` hook** — React hook that subscribes to language changes

No JSON files, no runtime parsing, no external libraries. All translations are embedded TypeScript. At ~100 keys x 10 languages, total size is well under 30KB.

## Key Structure

Keys are nested by feature area:

```
t.common          — shared labels (save, cancel, loading, error)
t.reports         — report list & cards (found(n), noReports, rating names)
t.detail          — ReportDetailModal (apply, edit, upvote, clear, sections)
t.editReport      — EditReportModal (labels, placeholders, install status)
t.settings        — SettingsTab (language, protonVersion, toggles)
t.compatTools     — compat tools tab (install, uninstall, otherVersion, zip)
t.configure       — ConfigureTab (edited reports, export)
t.toast           — toast messages (installed, failed, cleared, voted)
t.ratings         — platinum/gold/silver/bronze/borked/pending
```

Each leaf is either a `string` or a function (e.g., `(n: number) => string`) for dynamic values and pluralization.

## Dynamic Values and Pluralization

Keys needing variables are typed as functions:

```typescript
// TranslationTree type (excerpt)
reports: {
  found: (count: number) => string;
  noReports: string;
}

// English
reports: {
  found: (n) => n === 1 ? '1 report found' : `${n} reports found`,
  noReports: 'No reports found',
}

// Russian (complex pluralization)
reports: {
  found: (n) => {
    if (n % 10 === 1 && n % 100 !== 11) return `${n} отчёт найден`;
    if ([2,3,4].includes(n % 10) && ![12,13,14].includes(n % 100)) return `${n} отчёта найдено`;
    return `${n} отчётов найдено`;
  },
  noReports: 'Отчёты не найдены',
}
```

Compile-time safety: TypeScript enforces that every language implements the same function signatures. You cannot forget an argument.

## Detection Chain

When language preference is `'auto'` (default):

1. **Steam UI language** — `SteamClient.Settings.GetCurrentLanguage()` mapped to supported locale codes. Known mappings: `german` → `de`, `schinese` → `zh-CN`, `russian` → `ru`, `brazilian` → `pt-BR`, `spanish` → `es`, `french` → `fr`, `japanese` → `ja`, `koreana` → `ko`, `turkish` → `tr`. Unmapped Steam languages fall through to step 2.
2. **Browser language** — `navigator.language` matched to closest supported locale
3. **Fallback** — `'en'`

## Storage

The language preference is stored as `language: 'auto' | Language` in the existing plugin settings object alongside other plugin settings. Default: `'auto'`.

## Settings UI

A single `DropdownItem` in the existing SettingsTab:

- First option: `Auto (detected: Deutsch)` — shows the auto-detected language
- Followed by all 10 languages listed by native name

Changing the selection immediately re-renders the plugin UI. No restart required.

## Fallback Behavior

When a key exists in English but is missing from the active language:

- **Production**: English string returned silently
- **Dev builds**: English string prefixed with `[!]` and `logFrontendEvent('WARNING', 'Missing translation', { lang, key })` called

This ensures users always see usable text while developers can spot translation gaps during testing and in logs collected from the Deck.

## Component Integration

```typescript
import { t } from '../lib/i18n';

// Static string
<DialogButton>{t().detail.apply}</DialogButton>

// Dynamic value
<span>{t().reports.found(3)}</span>
```

`t()` is a function call so it always reads the current language. Internally it resolves the active language's tree with key-by-key English fallbacks applied.

**Re-rendering**: `setLanguage()` updates the stored preference and bumps a module-level React state signal. The `useLanguage()` hook in SettingsTab and the root plugin component subscribes to changes. Since language switches are rare (explicit user action), a full plugin tree re-render is acceptable.

## Migration Path

1. Create `src/lib/i18n.ts` with the full English tree and the `t()` / `setLanguage()` / `useLanguage()` infrastructure
2. Add language dropdown to SettingsTab
3. Replace hardcoded strings file-by-file, starting with the most visible surfaces (ReportDetailModal, ReportCard, SettingsTab)
4. Add remaining 9 language translations
5. Backfill less-visible surfaces (toasts, error messages, edge cases)

The English tree is the canonical complete set. TypeScript enforces that all other language trees match the same interface shape, catching missing keys at compile time.
